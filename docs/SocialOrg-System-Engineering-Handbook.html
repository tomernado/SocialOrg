פרק 01

סיכום מנהלים

מה זה SocialOrg?

SocialOrg הוא אגרגטור חדשות אישי מבוסס AI. המערכת בולעת פידי RSS מ-30+ מקורות חדשות נבחרים, משתמשת במודל השפה Gemini של גוגל כדי לסווג, לדרג ולסכם כל כתבה לתקציר עברי מובנה, ומאחסנת את התוצאה במסד נתונים SQLite מקומי שמוגש דרך REST API לממשק React.

ערך הליבה הוא אוצרות תוכן בקנה מידה רחב ללא מאמץ ידני: משימה תזמונית רצה כל שעה, משיכה ועיבוד אוטומטיים של תוכן חדש, כך שהמשתמש תמיד פותח את האפליקציה לפיד טרי ומסודר.

💡 SocialOrg אינה זרם בזמן אמת. זהו דייג׳סט מרענן-אצוות: כתבות מעובדות במחזורים שעתיים ונשמרות בצורה עמידה. ממשק המשתמש קורא מ-SQLite, לא מפידי RSS חיים. החלטה מכוונת — מנתקת בין זמינות הפיד לביצועי ממשק המשתמש, ומפשטת את מניעת הכפילויות.

שלושת פילרי הליבה

פילר 1 — אמינות (Reliability)

כל כתבה עוברת דרך צינור עמיד לכשלים. פיד פגום אחד, כתובת מתה, או כשל ב-Gemini API לא יכולים לקרוס את האצווה. כל כשל נתפס, מתועד וחוזר לשמירת תוכן גולמי. המערכת מסיימת כל ריצה ומדווחת: processed=N skipped=N errors=N.

פילר 2 — אוטומציה מבוססת AI

Gemini מבצע ארבעה תפקידים לכל כתבה: מסווג אותה לקטגוריית DB בעברית, מקצה תת-קטגוריה, כותב תקציר עברי (3 משפטים שלמים), ונותן ציון רלוונטיות מ-1 עד 10. זה אינו צ'אטבוט — זהו צינור פלט מובנה שבו ה-AI ממלא חוזה JSON מוקלד שנאכף על ידי Zod.

פילר 3 — מדרגיות עם מודעות לעלויות

המערכת רצה על 100% tier חינמי (Vercel Hobby, Render Free, Gemini Free). כל החלטת עיצוב — תור FIFO, עיכוב 5 שניות, מפסק-קצבה, מתג Turbo Mode — קיימת כדי למקסם תפוקה מבלי לשלם חשבון. המערכת מתוכננת לגדול בחן על ידי שדרוג צוואר הבקבוק הספציפי בלבד.

פרק 02

תשתית ומחסנית פריסה

Vercel (פרונטאנד) מול Render (בקאנד)

SocialOrg משתמשת בארכיטקטורת אירוח מפוצלת: ממשק ה-React הסטטי מוגש על ידי Vercel וה-API מבוסס Node.js מתארח ב-Render. אלו שני פרדיגמות אירוח שונות שנבחרו מסיבות שונות.

היבט Vercel (פרונטאנד) Render (בקאנד + DB)
מה הוא מגיש חבילת HTML/CSS/JS סטטית (פלט בנייה של React/Vite) REST API (Express) + קובץ SQLite
מודל ריצה Edge CDN — ללא שרת, קבצים מוגשים מ-40+ נקודות נוכחות תהליך Node.js מתמשך עם דיסק מחובר
טריגר פריסה push לגיט ← בנייה אוטומטית ← פריסה ל-CDN push לגיט ← הורדת קוד ← הפעלה מחדש של Node.js
מצב / עמידות חסר-מצב — ללא דיסק, ללא DB קובץ SQLite נשמר בין פריסות
למה הבחירה הזו CDN ללא הגדרות ל-SPA של React; HTTPS חינמי; פריסות preview אוטומטיות צריך filesystem עמיד ל-SQLite; מריץ את מתזמן cron
הגדרה קריטית משתנה סביבה VITE_API_BASE_URL שמצביע ל-Render משתנים ENABLE_AI_AGENT, GEMINI_API_KEY, FRONTEND_URL
⚠️ CORS הוא קריטי: מכיוון שהפרונטאנד והבקאנד נמצאים בדומיינים שונים (vercel.app מול onrender.com), כל תגובות ה-API חייבות לכלול את הכותרת Access-Control-Allow-Origin הנכונה. הבקאנד מסיר סלאש עוקב מ-FRONTEND_URL לפני הגדרת CORS — באג שגרם לכשל ייצור קשה לאבחון בעבר.

GitHub — בקרת קוד מקור ו-CI

GitHub משמש כמקור האמת היחיד וכטריגר הפריסה. גם Vercel וגם Render מחוברים למאגר GitHub ומאזינים לפוש לענף master. כשקומיט נוחת:

  1. Vercel מזהה את הפוש, מריץ npm run build בתוך frontend/, ופורס את החבילה הסטטית החדשה ל-CDN שלו תוך ~30 שניות.
  2. Render מזהה את אותו הפוש, מושך את הקוד החדש, מתקין תלויות, ומפעיל מחדש את תהליך Node.js תוך ~2 דקות.

אין שלב פריסה ידני. git push = פריסה לייצור. מכוון — מסיר שגיאת אנוש מתהליך הפריסה ומבטיח שהמאגר תמיד משקף בדיוק את מה שרץ בייצור.

UptimeRobot — פועם לב המערכת

ה-tier החינמי של Render מכבה שירותי אינטרנט לא פעילים לאחר 15 דקות של חוסר פעילות. הפעלה קרה לוקחת 30–60 שניות, מה שיגרום לקריאת ה-API הראשונה אחרי חוסר פעילות להרגיש שבורה למשתמש. UptimeRobot פותר זאת.

UptimeRobot מוגדר לפנג את נקודת הקצה לבדיקת בריאות של בקאנד Render (/health) כל 5 דקות. זה שומר על החם התהליך בכל עת. הוא גם משמש כטריגר cron חיצוני: UptimeRobot מוגדר בנוסף לקרוא POST /api/cron/fetch לפי לוח זמנים, ומשמש כפועם הלב החיצוני שמניע אינדוקסיית כתבות.

למה לא להסתמך רק על node-cron?

הבקאנד אכן מריץ node-cron פנימית, אבל cron מופעל רק אם התהליך חי. אם Render מכבה את התהליך (בשל חוסר פעילות) ואין בקשות נכנסות, ה-cron הפנימי לעולם לא מופעל. UptimeRobot הוא השומר חיצוני שמבטיח שבקשה תגיע לפני שפג חלון ההשבתה, שומר על חיות התהליך ועל אמינות ה-cron.

פרק 03

הצינור האג'נטי — נשמת הפרויקט

צינור האינדוקסיה ב-rss.service.js מדוגמן ארכיטקטורית כשלושה סוכנים נפרדים, כל אחד אחראי על תחום ברור. הפרדה זו הופכת את המערכת לניתנת לדיבוג, החלפה וביקורת.

סוכן A

בקר המקורות

הבעלים של מערך RSS_FEEDS — רישום המקורות הסמכותי. מצהיר על url, source_name, forceCategory, forceSubCategory, forceTag ו-isHebrew לכל פיד. תפקידו להגדיר אילו מקורות קיימים ולאכוף עקיפות מטא-דאטה קשות כך שה-AI לעולם לא יסווג פיד בצורה שגויה.

סוכן B

מפקח העיצוב

הבעלים של סכמת Zod (articleLlmSchema ב-ai.service.js) ומפעל buildPrompt(). תפקידו לאמת שכל תגובת AI נכונה מבנית לפני שנתון כלשהו נוגע במסד הנתונים. אוכף טיפוסים, טווחים (relevance_score 1–10) ואורכים מינימליים. תגובת AI פגומה זורקת — לעולם אינה משחיתת את ה-DB בשקט.

סוכן C

מנגנון הצינור

הפונקציה fetchAndParseFeeds() עצמה. מתזמרת את לולאת for...of החיצונית, מנהלת את דגל מפסק-הקצבה, מחליטה על מצב Turbo מול AI, קוראת להכנסה ל-DB ולגיזום FIFO, ותופסת כל מחלקת כשל בנפרד. אינווריאנט יחיד: האצווה תמיד חייבת להסתיים, ללא קשר לכשלים בודדים.

עיבוד טורי: למה for...of ולא Promise.all

זו אחת מהחלטות הארכיטקטורה החשובות ביותר בכל הקוד. הצינור מעבד כתבות אחת-אחת בלולאת for...of, לעולם לא במקביל. הסיבות:

היבט Promise.all (מקביל) for...of (טורי) ✓
מהירות הכי מהיר — כל הבקשות נורות בו-זמנית איטי יותר — כתבה אחת בכל פעם
הגבלת קצב חורג מיד ממגבלת 15 RPM של Gemini ← שגיאות 429 בכל ריצה עיכוב 5s לכתבה = 12 RPM — בטוח מתחת לסף
בידוד כשלים כשל אחד יכול להתפשט ולבטל בקשות אחרות בתעופה כל כתבה מבודדת לחלוטין — קריסה ב-#3 לעולם אינה משפיעה על #4
מפסק-קצבה בלתי אפשרי — כל הקריאות כבר נשלחו כשמגיעה שגיאת הקצבה הדגל מתהפך בשגיאה הראשונה; כל הכתבות הבאות מדלגות על Gemini
עקביות DB כתיבות מקביליות ל-SQLite (שהוא כותב-יחיד) גורמות לשגיאות נעילה כתיבות טוריות — מודל הכותב-היחיד של SQLite לעולם אינו נלחץ
ניתנות לדיבוג לוגים מעורבבים קשים מאוד לקריאה ומתאם לוגים כרונולוגיים וניתנים לעקיבה מושלמת לכל כתבה
✅ אמינות > מהירות הוא העקרון המנחה. הצינור רץ על cron — המשתמשים לא מחכים בזמן אמת לסיומו. עוד 10 דקות לעיבוד אצווה שלמה לא עולה כלום. אצווה שקרסה ובשקט מחקה 40 כתבות — זה כשל מוצר.

חילוץ תמונות — אסטרטגיה רב-שלבית

פידי RSS מסתירים כתובות URL של תמונות ב-5+ מיקומים שונים בהתאם לפורמט הפיד של המפרסם. הפונקציה extractImage() מיישמת אסטרטגיית מפל מסודרת לפי עדיפות: 

// שרשרת עדיפויות — ה-URL הראשון שאינו חסום מנצח
const candidates = [
  cleanUrl(item.enclosure?.url),         // ① enclosure סטנדרטי ב-RSS
  mediaUrl(item['media:content']),       // ② מרחב שמות podcast / media
  mediaUrl(item['media:thumbnail']),     // ③ thumbnail של YouTube
  imgFromHtml(item['content:encoded']), // ④ גוף HTML מלא (TechCrunch)
  imgFromHtml(item.content),              // ⑤ שדה תוכן חלקי
  imgFromHtml(item.description),          // ⑥ HTML של תיאור הפיד
];
return candidates.find((url) => url && !isBlockedImage(url)) ?? null;

הרג'קס BLOCKED_IMAGE_RE מסנן כתובות URL של placeholder/לוגו (תבניות כמו default, placeholder, blank) כך שלוגואים גנריים של מפרסמים לעולם לא יופיעו כתמונות כתבה. אם כל 6 האסטרטגיות נכשלות, הפרונטאנד מרנדר תמונת SVG scene ייעודית לקטגוריה.

פרק 04

ממשל AI ואופטימיזציית עלויות

ממשל AI הוא המשמעת של הבטחה שמערכת AI מתנהגת בצורה צפויה, בתוך תקציב, עם התנהגות fallback מוגדרת כאשר היא נכשלת. SocialOrg מיישמת ארבעה מנגנוני ממשל.

מצב Turbo מול Production AI

לצינור שני מצבי פעולה, שנשלטים על ידי משתנה סביבה אחד: ENABLE_AI_AGENT=true|false שמוגדר בדשבורד של Render. ללא שינוי קוד, ללא פריסה מחדש כדי לעבור ביניהם.

מאפיין 🚀 Turbo Mode (false) 🤖 Production AI (true)
Gemini נקרא? לא — מדולג לחלוטין כן — לכל כתבה חדשה
עיכוב 5 שניות? לא — אצווה מיידית כן — הגבלת קצב נאכפת
תוכן ai_summary contentSnippet גולמי מ-RSS או כותרת תקציר עברי 3 משפטים שנוצר על ידי Gemini
relevance_score מקודד קשה 10 ציון Gemini 1–10
קצבת API בשימוש? אפס ~250 טוקני פלט לכתבה
מהירות אצווה שלמה ב-<30 שניות אצווה שלמה ב-10–20 דקות
מקרה שימוש פיתוח, בדיקות, התאוששות מקצבה, אכלוס DB ראשוני פעולת ייצור רגילה
סמן לוג [TURBO] [AI]
💡 היכולת לעבור מצבים דרך משתנה סביבה — מבלי לגעת בקוד או לטריגר פריסה — היא עקרון תפעול ייצור מקצועי. מאפשר תגובה מיידית לתשישות קצבה ללא כל עבודת פיתוח.

מפסק-הקצבה (isQuotaExhausted)

זהו מנגנון האמינות המתוחכם ביותר בכל הקוד. הבעיה המדויקת שהוא פותר:

ל-tier החינמי של Gemini יש מכסת בקשות יומית. אם הצינור מעבד 50 כתבות וכתבה #12 מטריגרת תשישות קצבה, כתבות #13–50 יבצעו כל אחת קריאת API, יקבלו שגיאת 429, יתעדו כשל ויתקדמו — מבזבזים 38 קריאות API מיותרות, 38 × עיכוב 5 שניות (3+ דקות המתנה מיותרת), ו-38 רשומות שגיאה שמסתירות את הבעיה האמיתית.

המפסק-קצבה מונע את כל זה:

// מוצהר בתוך fetchAndParseFeeds() — מתאפס בכל הפעלה
let isQuotaExhausted = false;

// בבלוק ה-catch:
const isQuotaError =
  err.message.startsWith('QUOTA_EXHAUSTED:') ||
  err.message.includes('429')             ||
  err.message.toLowerCase().includes('quota');

if (isQuotaError && !isQuotaExhausted) {
  isQuotaExhausted = true;  // היפוך הדגל — פעם אחת בלבד
  console.warn('[ai] ⚡ QUOTA EXHAUSTED — עובר למצב Turbo...');
}

ברגע שהדגל הוא true, נתיב הקוד בראש לולאת הכתבה מזהה אותו:

if (isQuotaExhausted) {
  // שמירה מיידית כ-Turbo — ללא קריאת Gemini, ללא עיכוב
  const article = { ..., relevance_score: 0, ai_summary: extractSummary(item, title) };
  const inserted = insertArticle(db, article);
  if (inserted) { console.log(`[db] ✓ [QUOTA-TURBO] "${title}"...`); }
}
למה להצהיר על הדגל בתוך הפונקציה (ולא בהיקף המודול)?

אם isQuotaExhausted היה מוצהר בהיקף המודול (מחוץ לפונקציה), הוא היה נמשך בין הפעלות. ריצת cron השעתית הבאה הייתה מתחילה עם הדגל עדיין true — כלומר התאוששות הקצבה (Gemini מתאפס כל שעה) לעולם לא תנוצל. על ידי הצהרתו בתוך fetchAndParseFeeds(), כל הפעלה חדשה מתחילה עם false טרי. אלו הן הסמנטיקות "לכל ריצה".

הגבלת קצב — עיכוב 5 שניות

ה-tier החינמי של Gemini Flash Lite אוכף מגבלה של 15 בקשות לדקה (RPM). הצינור מכניס await sleep(5000) של 5 שניות לפני כל קריאת Gemini.

החשבון:

  • 60 שניות ÷ 5 שניות = 12 בקשות לדקה
  • 12 RPM הם 80% מסף 15 RPM, משאירים מרווח של 3 בקשות לניסיונות חוזרים
  • לוגיקת הניסיון החוזר המובנית ב-callWithRetry() קוראת את כותרת retry-after של ה-API וממתינה בדיוק את הזמן המומלץ לפני הניסיון השני
🚫 אל תנמיך את העיכוב מתחת ל-4 שניות. ב-4 שניות, אתה בדיוק ב-15 RPM — ניסיון חוזר אחד בכל מקום באצווה ידחוף אותך מעל. ה-5 שניות הנוכחיות מספקות מרווח. ב-3 שניות (20 RPM) תקבל שגיאות 429 שיטתיות בכל ריצה.

חוזה כללי ה-AI

ה-prompt ב-buildPrompt() אינו הוראה חופשית — הוא חוזה התנהגותי שה-AI חייב לציית לו. לכל כלל ב-prompt יש סיבה הנדסית ספציפית:

כלל ב-Prompt למה הוא קיים מה נשבר אם מסירים אותו
"כתוב בדיוק 3 משפטים שלמים" מבטיח אורך תקציר עקבי בין כל הכתבות AI מחזיר משפט אחד לחדשות פשוטות, 8 משפטים למורכבות — ממשק משתמש לא עקבי
"כל משפט חייב להסתיים בנקודה" מאפשר ל-cleanSummary() לחתוך למשפט שלם אחרון בצורה אמינה תקצירים עלולים להסתיים באליפסיס, באמצע משפט, או עם שיירי Markdown
"החזר JSON תקין בלבד" מונע עטיפת Markdown (```json ... ```) JSON.parse() זורק בכל תגובה; כל הכתבות חוזרות לתוכן גולמי
הזרקת forceSubCategory מבטיח שה-AI משתמש בדיוק בתת-הקטגוריה שהפיד מצהיר עליה AI עלול לסווג כתבת ברצלונה כ-FOOTBALL-NEWS; מופיעה בטאב הלא נכון
responseMimeType: 'application/json' מפעיל את מצב הפלט המובנה של Gemini ברמת ה-API בלעדיו, Gemini מתייחס לבקשה כהשלמת שיחה ומוסיף פרוזה מסביב ל-JSON
temperature: 0.1 טמפרטורה קרובה לאפס = פלט כמעט דטרמיניסטי טמפרטורה גבוהה יותר מכניסה הלוצינציות יצירתיות בשדות מובנים

ממשל נתונים

לוגיקת תור FIFO (מקסימום 10 כתבות למקור)

ללא מגבלה, מסד הנתונים יגדל ללא גבולות. מקור בנפח גבוה אחד כמו BBC יכול לפרסם 100 כתבות ביום, ובסופו של דבר לדחוק החוצה כתבות ממקורות קטנים יותר ומאוצרים יותר כמו Fabrizio Romano.

גיזום FIFO (ראשון נכנס, ראשון יוצא) מוחל לפי זוג (sub_category, source_name) אחרי כל הכנסה מוצלחת. המגבלה היא 10 כתבות לזוג. 

// schema.js — trimSubCategory()
const MAX = 10;

const count = db
  .prepare(`SELECT COUNT(*) as n FROM articles
     WHERE sub_category = ? AND source_name = ?`)
  .get(subCategory, sourceName).n;

if (count <= MAX) return;  // אין מה לגזום

db.prepare(`DELETE FROM articles WHERE id IN (
  SELECT id FROM articles
  WHERE sub_category = ? AND source_name = ?
  ORDER BY created_at ASC  -- הישנים ביותר ראשונים
  LIMIT ?
)`).run(subCategory, sourceName, count - MAX);
למה לזוג על (sub_category + source_name) ולא רק sub_category?

אם היינו גוזמים ברמת תת-הקטגוריה, BBC Sport יכול לאכלס את כל 10 חריצי NATIONAL, ולא יישאר מקום ל-Sky Sports או לכל פיד לאומי אחר. הזיווג למקור נותן לכל מקור תור עצמאי של 10 חריצים. BBC ו-Sky Sports מתחרים בתוך התורות שלהם, לא אחד מול השני.

מניעת כפילויות לפני קריאות LLM

הפעולה הראשונה ביותר בתוך לולאת הכתבה — לפני כל קריאת Gemini, לפני כל חילוץ תמונה, לפני כל עיבוד — היא בדיקת אידמפוטנטיות: 

if (articleExists(db, url)) {
  result.skipped++;
  continue;  // אין קריאת API, אין עיבוד
}

בדיקה יחידה זו מספקת שלושה ערבויות:

  • שליטת עלויות: כתבה שהופיעה באצווה של אתמול לעולם אינה מסוכמת מחדש. כל כתבה עולה בדיוק קריאת Gemini אחת בכל חייה.
  • אידמפוטנטיות: קריאה לצינור האחזור 10 פעמים ברצף מייצרת את אותו מצב DB כמו קריאה אחת. בטוח לטריגר ידנית בכל עת.
  • מהירות: בריצה טיפוסית שבה רוב הכתבות כבר ב-DB, 90%+ מגוף הלולאה הוא בדיקת SELECT 1 פשוטה — מיקרו-שניות לכתבה.

מסד הנתונים אוכף זאת ברמת הסכמה עם אילוץ UNIQUE על original_url. גם אם לבדיקת הדה-דופ יש מצב תחרות (שתי ריצות צינור מקביליות), משפט INSERT OR IGNORE מבטיח שהמסד תמיד נשאר עקבי.

אימות Zod — לעולם אל תסמוך על טקסט גולמי מ-LLM

כל תגובת Gemini מנותחת כ-JSON ולאחר מכן מאומתת מול סכמת Zod קפדנית לפני שנתון כלשהו נוגע במסד הנתונים. זהו כלל איכות נתונים שאי אפשר לסחור בו. 

const articleLlmSchema = z.object({
  category: z.enum([
    'עולם ה-AI והפיתוח',
    'חדשות בעולם הספורט',
    'חדשות בעולם הכדורגל',
    'חדשות כלליות בעולם הבידור',
    'חדשות כלליות בעולם',
  ]),
  sub_category: z.string().min(1),
  tag: z.string().min(1),
  relevance_score: z.number().int().min(1).max(10),
  ai_summary: z.string().min(10),
});

אם Gemini מחזיר category שאינו ב-enum המותר, Zod זורק. בלוק ה-catch שומר fallback גולמי. מסד הנתונים לעולם אינו מקבל כתבה לא מסווגת או פגומה. זו הערבות המרכזית: סכמת ה-DB היא החוזה, Zod הוא המאכף.

מה Zod תופס דוגמת כשל AI ללא Zod
קטגוריה לא חוקית AI מחזיר "Sports News" במקום "חדשות בעולם הספורט" כתבה נשמרת עם קטגוריה שגויה, מופיעה בטאב הלא נכון
ציון מחוץ לטווח AI מחזיר relevance_score: 11 או 0 תג ה-UI מציג "11/10" או ריק; לוגיקת המיון נשברת
תקציר ריק AI מחזיר ai_summary: "" או מחסיר את השדה כרטיס הכתבה מציג תקציר ריק; המודל מציג כלום בבלוק התוכן
שדות נוספים הלוצינציות AI מוסיפות "confidence": 0.9 ל-JSON שדה נוסף מתעלם בשקט ב-JS, אך מאותת על סחרור ב-prompt

עיצוב סכמת מסד הנתונים

CREATE TABLE IF NOT EXISTS articles (
  id              TEXT PRIMARY KEY,           -- UUID v4
  original_url    TEXT NOT NULL UNIQUE,       -- מפתח מניעת כפילויות
  title           TEXT NOT NULL,
  image_url       TEXT,                       -- nullable: SVG fallback בפרונטאנד
  category        TEXT NOT NULL DEFAULT 'Uncategorised',  -- ערך בעברית
  sub_category    TEXT NOT NULL DEFAULT 'Uncategorised',  -- קבוע באנגלית
  source_name     TEXT NOT NULL DEFAULT '',
  tag             TEXT NOT NULL DEFAULT 'General',
  relevance_score INTEGER NOT NULL DEFAULT 0,
  ai_summary      TEXT NOT NULL DEFAULT '',
  created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

-- אינדקס על תבנית השאילתא הנפוצה ביותר
CREATE INDEX IF NOT EXISTS idx_articles_sub_category_created
  ON articles (sub_category, created_at);

האינדקס המורכב על (sub_category, created_at) הוא אבן הביצועים של כל שאילתת API. הפרונטאנד תמיד מסנן לפי קטגוריה או תת-קטגוריה, ממוין לפי זמן. בלי אינדקס זה, SQLite יבצע סריקת טבלה מלאה בכל טעינת עמוד.

תבנית מיגרציית סכמה

ל-SQLite אין ALTER TABLE ... ADD COLUMN IF NOT EXISTS. כשצריך עמודה חדשה, הגישה האידיומטית הבטוחה היא try/catch סביב ה-ALTER: 

// רץ בכל הפעלה — בטוח לקרוא פעמים רבות
try {
  db.exec(`ALTER TABLE articles ADD COLUMN source_name TEXT NOT NULL DEFAULT ''`);
} catch {
  // העמודה כבר קיימת — SQLite זורק, אנחנו מתעלמים
}

למיגרציות נתונים (שינוי שמות ערכים בשורות קיימות), נעשה שימוש במשפטי UPDATE פשוטים — הם אידמפוטנטיים באופן טבעי מכיוון שאחרי הריצה הראשונה, אין שורות שתואמות את תנאי WHERE הישן: 

db.exec(`
  UPDATE articles SET sub_category = 'NATIONAL' WHERE sub_category = 'NATIONAL-MONDIAL';
  UPDATE articles SET sub_category = 'FOOTBALL' WHERE sub_category = 'PREMIER-LEAGUE';
  UPDATE articles SET sub_category = 'FOOTBALL' WHERE sub_category = 'TRANSFERS';
`);

תחזוקה ועמידות לעתיד

איך להוסיף פיד RSS חדש — שלב אחרי שלב

  1. החליט על הקטגוריה ותת-הקטגוריה. עיין ב-CATEGORY_MAP ב-frontend/src/App.jsx לחמשת שמות הקטגוריות האנגליים החוקיים, וב-CATEGORY_SUB_OPTIONS לקבועי תת-הקטגוריה החוקיים. אם צריך תת-קטגוריה חדשה, חייב להוסיף אותה גם ל-CATEGORY_SUB_OPTIONS (פרונטאנד) וגם ל-SUB_CATEGORY_LABELS (ArticleCard.jsx), ואופציונלית להוסיף ויזואל fallback ל-SUB_CATEGORY_FALLBACK.
  2. מצא או בנה את כתובת ה-RSS. לרוב אתרי החדשות יש נקודת קצה /feed, /rss, או /rss.xml. אם האתר חוסם RSS ישיר או אין לו פיד, השתמש ב-FetchRSS.com כדי לייצר כתובת פיד מדוממת. בדוק את הכתובת בדפדפן או עם curl תחילה.
  3. הוסף את הגדרת הפיד ל-RSS_FEEDS ב-rss.service.js: 
    {
  url:             'https://example.com/rss.xml',
  source_name:     'המקור שלי',         // מוצג בכרטיסי UI ובמודלים
  forceCategory:   'AI & Tech',         // חייב להתאים ל-CATEGORY_MAP בדיוק
  forceSubCategory:'GADGETS-LEAKS',     // חייב להתאים ל-CATEGORY_SUB_OPTIONS
  forceTag:        '#גאדגטים',          // האשטאג עברי שמוצג על כרטיס הכתבה
  isHebrew:        true,               // אופציונלי: מדלג על רמז תרגום ב-prompt
}
  4. בדוק במצב Turbo תחילה. הגדר ENABLE_AI_AGENT=false ב-Render, טריגר אחזור ידני דרך POST /api/trigger-fetch, ואמת שכתבות מהמקור החדש מופיעות בטאב ה-UI הנכון. חזור למצב AI רק לאחר שהפיד אושר.
  5. בצע commit ו-push. Vercel וגם Render יפרסו אוטומטית. אם תת-הקטגוריה חדשה, הפרונטאנד נפרס ראשון (כפתור חדש מופיע ב-UI) — כתבות עלולות להופיע עם עיכוב קל בזמן ש-Render מופעל מחדש וה-cron הראשון מופעל.

יישום כללי ממשל אלה בפרויקטים חדשים

SocialOrg ביסס סט תבניות ארכיטקטורה שחלות על כל צינור מבוסס LLM. הנה רשימת הבדיקה המזוקקת:

כלל ממשל איך ליישם בכל פרויקט
לעולם אל תסמוך על פלט גולמי מ-LLM תמיד נתח JSON + אמת עם ספריית סכמה (Zod, Pydantic וכו') לפני שמירה. התייחס ל-LLM כ-API חיצוני לא מהימן — בדיוק כפי שתאמת קלט משתמש.
עיבוד טורי + מבודד לצינורות אצווה שקוראים ל-API עם הגבלת קצב, השתמש בלולאות טוריות עם try/catch לכל פריט. לעולם אל תשתמש ב-Promise.all — API עם הגבלת קצב ועיבוד מקבילי אינם תואמים.
מפסק-קצבה לכל ריצה הצהר על דגלי קצבה בתוך פונקציית האצווה, לא בהיקף המודול. כשה-API מאותת על תשישות, הפסק לקרוא לו והשלם את האצווה באמצעות נתיב fallback. אפס בהפעלה הבאה.
מניעת כפילויות לפני קריאות LLM תמיד בדוק רשומות קיימות לפני קריאת ה-AI. מונע תשלום כפול על אותו תוכן והופך את הצינור לאידמפוטנטי — בטוח לנסות שוב בכל עת.
עקיפה קשה מעל סיווג AI למטא-דאטה שאתה שולט בה (קטגוריות, תגיות, מקורות), קודד אותה קשה כעקיפות בהגדרות הצינור שלך. לעולם אל תתן ל-AI להקצות שדות סיווג קריטיים ללא בדיקה קשה.
מתג מצב דרך משתנה סביבה הוסף מתג בסגנון ENABLE_AI מהיום הראשון. כשה-API שובר או הקצבה תשושה, אתה צריך להיות מסוגל להמשיך להריץ את המערכת ללא פריסת קוד.
